title: 'Multi-LiDAR setup'
description: 'Connect and configure multiple LiDAR devices simultaneously, including mixed HAP and Mid360 setups and multiple NIC scenarios.'

The driver supports running multiple LiDAR devices at the same time. There are two common scenarios:

- **Mixed models on one NIC** — HAP and Mid360 connected to a single network interface, configured in one launch.
- **Multiple NICs** — each NIC connects to one or more LiDARs; you run a separate driver instance (launch file) per NIC.

## Scenario 1: HAP + Mid360 on a single NIC

Use `mixed_HAP_MID360_config.json` and `rviz_mixed.launch` (ROS1) or `rviz_mixed.py` (ROS2) to launch the driver.

The mixed config file contains a section for each LiDAR model under `HAP` and `MID360`, plus one entry per device in the `lidar_configs` array.

```json config/mixed_HAP_MID360_config.json
      "cmd_data_port": 56000,
```

</Note>

### Launch commands

<CodeGroup>

```bash ROS1 — mixed HAP + Mid360
source ../../devel/setup.sh
roslaunch livox_ros_driver2 rviz_mixed.launch
```

```bash ROS1 — mixed, custom message format
source ../../devel/setup.sh
roslaunch livox_ros_driver2 msg_mixed.launch
```

```bash ROS2 — mixed HAP + Mid360
source ../../install/setup.sh
ros2 launch livox_ros_driver2 rviz_mixed.py
```

</CodeGroup>

## Scenario 2: Multiple NICs

When each LiDAR is connected via a separate network interface, you need one config file and one launch file per NIC. Each driver instance communicates with its own LiDAR over the matching interface.

<Steps>
  <Step title="Create a config file per NIC">
    Create `MID360_config1.json` for the first NIC (`192.168.1.x`) and `MID360_config2.json` for the second (`192.168.2.x`). The `host_net_info` array format lets you bind each LiDAR IP to a specific host IP.

    <Tabs>
      <Tab title="MID360_config1.json">

        ```json config/MID360_config1.json
              "log_data_port": 56500

            "host_net_info": [
              {
                "lidar_ip": ["192.168.1.100"],
                "host_ip": "192.168.1.5",
                "cmd_data_port": 56101,
                "push_msg_port": 56201,
                "point_data_port": 56301,
                "imu_data_port": 56401,
                "log_data_port": 56501
        ```

    Each launch file references its own config file via `user_config_path` and uses a unique node name to avoid conflicts.

    <Tabs>
      <Tab title="Launch1 (NIC 1)">
        ```xml launch_ROS1/msg_MID360_nic1.launch

          <param name="xfer_format"      value="$(arg xfer_format)"/>
          <param name="multi_topic"      value="$(arg multi_topic)"/>
          <param name="publish_freq"     type="double" value="$(arg publish_freq)"/>
          <param name="user_config_path" type="string"
                 value="$(find livox_ros_driver2)/config/MID360_config1.json"/>

          <node name="livox_lidar_publisher1" pkg="livox_ros_driver2"
                type="livox_ros_driver2_node" required="true"

        </launch>
        ```

## Per-device topics with `multi_topic`

By default, all LiDARs publish to a single shared topic. When you set `multi_topic=1`, each LiDAR gets its own topic. The topic name includes a suffix derived from the LiDAR's IP address, which lets you distinguish data from individual sensors downstream.

<Warning>
  Use `multi_topic=1` when you need to process data from each LiDAR independently. If all sensors feed into the same processing pipeline, the default `multi_topic=0` is simpler.
</Warning>

```bash ROS1 — enable per-device topics
roslaunch livox_ros_driver2 rviz_mixed.launch multi_topic:=1
```
